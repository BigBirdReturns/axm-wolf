Abstract

A capture pack is a portable JSON question library that defines a domain (sections, lenses, prompts, presentation). The engine runs it the same way every time, on any device, with no network. The subject answers in any order. The browser preserves the testimony locally. The record exports to durable, portable formats. No backend, no accounts, no LLM at runtime.

IWhat Wolf Is

Wolf captures tacit institutional knowledge from people who were present when consequential work, decisions, failures, and informal practices occurred. Traditional knowledge systems assume documentation happens during the work; it usually doesn’t. Wolf does retrospective capture: a structured question library helps someone recover memory, they answer asynchronously, and the browser preserves the testimony. Retirees. Founders after exit. Departing operators. Incident witnesses. People whose experience exists primarily in memory.

The interface is replaceable. The question library is not. A capture pack contains the domain work — what to ask, how to group prompts, what contextual cues recover memory, what metadata makes the resulting record intelligible later. The engine knows nothing about the domain. It validates the pack, stores responses, manages revisions, computes progress, runs search, builds bundles, renders exports. Load a different pack and you have a different deposition.

The pack is the question contract. If it passes validatePack(), it will run.

IIFour Guarantees

01
Append-only testimony
Editing a saved response creates a new revision; the prior revision is preserved and exportable. There is no overwrite mode. The raw response record is the source of truth — every derived view (progress, search, summaries) computes from it.
commitResponse · supersedes chain · revision history view
02
Inert packs
Imported packs are JSON, not code. The schema rejects HTML, JavaScript URLs, unknown fields, and prototype-pollution keys. All content renders through React text nodes — never innerHTML. Validation fails loudly; nothing partially installs.
validatePack · hand-rolled schema · bounded sizes
03
Content-free engine
No prompt text, subject name, or section label is hardcoded in engine or storage code. A boundary linter scans every commit. The first pack is bundled as data; the engine doesn’t know about William Sandhu any more than it knows about the next subject.
src/engine/ has zero imports from src/packs/ or src/app/
04
Offline-first
After first load, the app shell and bundled pack are precached. Capture, edit, search, import, and export work with the browser offline. Voice input is a progressive enhancement — labeled as browser- and network-dependent, never described as offline.

IIIThe First Record

The first bundled pack is The Wolf’s Deposition — a 40-year career in institutional IT, organized as a Pulp Fiction-inflected oral history of William Sandhu, framed as a deposition rather than an interview. Seven sections trace early years through HP, Agilent, EDS, Autonomy, and the aftermath. Fifty-eight prompts, each grouped under a lens (The Room, The Decision, The Person, The Moment, The Thing Nobody Said).

The legacy proof-of-concept claimed sixty-two questions. The source contained fifty-eight. We did not invent four to match the copy. The mismatch is documented in reference/legacy-v0/AUDIT.md and the test suite asserts the real counts so the lie cannot grow back. Every legacy era__index key has a checked-in migration to a semantic prompt ID; running Wolf on the same origin migrates the legacy answers into one record with imported revisions and a recovery report for anything unmapped.

The first pack is Wolf’s gold shard. It is preserved, not edited. A change to the engine that breaks its tests fails the gate.

VPortable Formats

Both formats are plain JSON. They round-trip losslessly: an exported bundle re-imported into a clean install rebuilds every revision, every draft (optional), every timestamp.

.wolfpack.json // portable capture pack schemaVersion packId, packVersion, engineVersion title, subtitle, description subjectDefaults theme { accent } lenses [] // e.g. The Room, The Decision sections [] promptIds [] prompts [] id, kind: "long_text" lensId text // plain text — no HTML context, tags, required exportDefaults .wolfrecord.json // portable record bundle schemaVersion recordId, title, subject, status pack { packId, packVersion, packDigest, snapshot } responses [] promptId revisions [] // append-only; full history drafts [] // optional, opt-in provenance { engineVersion, appVersion, exportedAt }

Every installed pack carries a SHA-256 packDigest over canonical JSON. The digest detects accidental or untracked changes; it does not prove authorship. Signed packs through AXM verification tooling (Genesis suites) are on the roadmap, not the present — Wolf must never present a digest as a signature.
